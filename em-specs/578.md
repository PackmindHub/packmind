# Remove a published plugin from a marketplace

## 1. Context & problem

Packmind is the curation and governance layer for an engineering team's playbook, and the distribution layer that renders that playbook into the formats each AI coding agent expects. When a team publishes a plugin (a standard, command, or skill bundled for a downstream marketplace such as `acme/claude-marketplace`), the publication crosses into Git through a governed PR — never a direct push.

Retirement, today, is not symmetric. A package can be removed in Packmind, but the corresponding artifact in the downstream marketplace repo is left to manual cleanup, or removed by hand on the Git side without Packmind ever knowing. Both paths break the governance promise: stale plugins linger in marketplaces, or marketplaces silently diverge from Packmind's intent.

This PRD closes the loop. Retirement becomes PR-governed (symmetric to publication), and any direct deletion in the marketplace repo is surfaced as drift in the governance view.

## 2. Goals & non-goals

**Goals**

- Let an org member retire a published plugin through the same PR-based governance flow that ships it.
- Make the marketplace's state honest: a plugin retired in Packmind is visibly "to be removed" until the deletion PR lands.
- Detect divergence when a plugin is deleted directly in the marketplace repo, and surface it as drift.

**Non-goals**

- Auto-merging deletion PRs — the human review step stays.
- Bulk retirement across multiple marketplaces in a single action.
- Restoring a previously deleted plugin from Packmind.
- Cleaning up usage of the retired plugin in repos that already installed it.

## 3. Personas & use cases

- **Org member managing a marketplace** — typically a tech lead or staff engineer per Packmind's primary user profile. They own the marketplace's content and decide what stays current. The story name uses "Cédric" as a concrete example.
- **Frequency / volume** — occasional, bursty: cleanup tends to happen after a standards refresh, a tool deprecation, or a marketplace audit.
- **Secondary personas** — reviewers of the deletion PR (other org members or repo CODEOWNERS) — they exercise the governance check but do not trigger the retirement.

## 4. User story

As an **org member**,
I want **to remove a published plugin from a marketplace**,
so that **I can retire deprecated content and keep the marketplace honest**.

## 5. Success metrics

- **Adoption**: number of plugins retired via the PR flow, per marketplace, per week.
- **Governance health**: ratio of retirements going through the PR flow vs. retirements detected only as drift — the higher the PR share, the healthier the governance.
- **Time-to-completion**: median time between "mark as to be removed" and deletion PR merged.

## 6. Functional requirements

### Rule: Removing a managed package opens a deletion PR (symmetric to publish)

Deletion is governed the same way publication is — through a PR against the marketplace repo, never a direct push.

**Scenario: Removing a package from the marketplace in Packmind opens a deletion PR**

- **Given** `@devops/security` plugin lives on `acme/claude-marketplace`
- **When** Cédric removes the package from the marketplace
- **Then** Cédric can open a PR to delete it from the repo

**Scenario: Deleting the original Packmind package marks the plugin as "to be removed"**

- **Given** `@devops/security` plugin lives on the repo
- **When** Cédric deletes the original package in Packmind
- **Then** it is marked as "to be removed" in the marketplace details
- **And** Cédric can open a PR to delete it

### Rule: Direct deletion in the marketplace repo is detected as drift

A retirement that bypasses Packmind (someone deletes the plugin file directly in the Git repo) must not silently break the governance state — it has to show up as drift.

**Scenario: Plugin deleted in the repo is detected as drift**

- **Given** `@devops/security` distributed by Packmind on `acme/claude-marketplace` was deleted directly in the repository
- **When** Packmind scans the repository content
- **Then** marketplace status changes to "Drift detected" with details

## 7. Non-functional requirements

- **Performance**: a marketplace drift scan completes within a few minutes for a marketplace holding up to ~hundreds of plugins; no synchronous user-blocking wait. _(inferred — confirm)_
- **Security & privacy**: deletion PR creation reuses the existing Packmind → Git auth surface (the same path used by publication) — no new permission scope. The actor who triggered the retirement is logged for audit. _(inferred — confirm)_
- **Accessibility**: the new "to be removed" state and the "Drift detected" badge use text + icon, not colour alone (WCAG 2.1 AA per `PRODUCT.md`). Keyboard reachability and focus state on the new "remove" action match existing marketplace actions. _(inferred — confirm)_
- **Observability**: every retirement step (`removal_initiated`, `pr_opened`, `marked_for_removal`, `drift_detected`) emits a structured log with `marketplace_id`, `plugin_slug`, `actor_id`, and the resulting state transition. _(inferred — confirm)_

## 8. Dependencies & risks

**Dependencies**

- Existing PR-based publication flow (auth, PR templating, status reporting). Retirement reuses the same pipeline.
- Existing marketplace scan / sync job that compares Packmind's intent to the Git repo content — this is what makes drift detection possible. _(inferred — confirm)_

**Risks & mitigations**

- Deletion PR is opened then abandoned (never merged): plugin stays in "to be removed" state indefinitely → surface stuck-state in the governance view (age of pending deletions), and let the actor cancel the retirement.
- Drift detection false positives during a scan window where the PR has merged but Packmind's cache is stale → tie the scan to the same source of truth used by publication, and de-duplicate drift events against an open deletion PR.
- A retirement that loses the PR link (PR closed without merge, repo renamed): Packmind cannot tell apart "abandoned" from "merged elsewhere" → require a `Closes` reference or rely on the marketplace scan's authoritative read. _(inferred — confirm)_

## 9. Rollout plan

- **Feature flag**: `flag.marketplace.plugin-removal` _(inferred — confirm)_ — gates both the new "remove" action and the new drift type. Default off until validated end-to-end.
- **Phasing**:
  1. Deletion PR flow (rule R-remove-flow) — ship behind the flag, dogfood on Packmind's own marketplaces.
  2. "To be removed" state surfaced in marketplace details — ship with phase 1.
  3. Drift detection for deleted-in-repo (rule R-remove-drift) — ship once phase 1 is stable; without it, drift is invisible but no regression vs. today.
- **Rollback**: flag flip turns off the new action and the new drift type; existing publication flow is untouched, no destructive cleanup needed.
- **Communication**: changelog entry under Unreleased, end-user doc updated under `apps/doc/`, internal announcement to tech leads using marketplaces.
