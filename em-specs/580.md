# Publish Packmind package as managed plugin on a linked marketplace

## 1. Context & problem

Packmind is the source of truth for an engineering team's playbook (standards, commands, skills) and the distribution layer that renders it for every AI coding agent in every repo. Teams already curate packages in Packmind; what is missing today is a path to publish them onto a Git-based agent marketplace (e.g. `acme/claude-marketplace`) as **managed plugins** that the marketplace's consumers can install like any other plugin, but that remain governed by Packmind.

Without this, packages either stay locked inside Packmind or get copy-pasted into marketplace repos by hand — exactly the drift the product exists to eliminate. The desired state: an org member curates a package once, publishes it, and the linked marketplace exposes it as a versioned, Packmind-owned plugin alongside the other plugins of that marketplace.

## 2. Goals & non-goals

**Goals**

- Any org member can publish a curated Packmind package to a linked marketplace and have it appear as a managed plugin consumable by that marketplace's users.
- Publishing is **traceable** end-to-end: every publish produces a PR on the marketplace repo carrying Packmind governance metadata.
- Republishing the same package converges to the same rolling PR — no PR explosion, no duplicate plugins in the marketplace descriptor.
- Publish fails fast and explicitly when contracts (descriptor present, name unique, token valid) are broken; the org member sees a precise reason.

**Non-goals**

- Linking a marketplace to an organization — out of scope; covered by the admin-only linking flow. Covered by #541
- Consuming / installing the published plugin from a target repo — covered by the install flow on consumer projects
- Multi-marketplace fan-out (publishing the same package to several marketplaces in one action).
- Unpublishing or removing a managed plugin from a marketplace. Covered by #578

## 3. Personas & use cases

Primary persona is an **org member** — typically a tech lead, staff engineer, or engineering manager (per PRODUCT.md) who curates the team's playbook in Packmind and wants to ship a package onto the team's marketplace. Unlike the **linking** flow, this action is **not** gated to org admins: any org member can publish, on the assumption that curation already happened upstream.

- **Org member (publisher)** — has access to at least one space holding curated packages; the org has at least one linked marketplace. They publish from the Packmind UI and expect the action to either complete with a PR link or fail with a clear, actionable error.
- **Org admin** — appears only as the persona who previously linked the marketplace; not the publisher in this PRD.
- **Marketplace consumer** — a developer in a target repo who installs the resulting managed plugin. Out of scope here but present as the downstream beneficiary.
- **Frequency / volume** — publishes happen many times per week per active org once a marketplace is linked; bursts on first onboarding and around package releases.

## 4. User story

As an **org member**,
I want to **publish a Packmind package as a managed plugin on a linked marketplace**,
so that **my team can consume it as a versioned, governed artifact**.

## 5. Success metrics

- **Activation**: % of orgs with at least one linked marketplace that publish their first package within 7 days of linking. \*
- **Adoption**: weekly count of successful publishes per active org; ratio of "rolling-PR amendments" vs "new PR opened" (signals the rolling-PR mechanic is working as designed).
- **Quality / health**: publish success rate; share of failures broken down by reason (descriptor missing, name conflict, token expired, other); p95 latency from "Publish" click to PR visible on the marketplace repo.
- **Amplitude events to track** _(inferred — confirm)_:

  - `plugin_publish_succeeded` — properties: `package_id`, `marketplace_slug`, `pr_url`, `was_noop`, `commit_count_after`
  - `plugin_publish_failed` — properties: `package_id`, `marketplace_slug`, `failure_reason` (one of `descriptor_missing`, `name_conflict_unmanaged`, `invalid_token`, `other`)

## 6. Functional requirements

### Rule: Publishing creates or amends a rolling PR with Packmind governance traces _(existing)_

**Scenario: Publish creates a PR on the marketplace repo**

- **Given** package "AWS management" in space "Devops"
- **And** marketplace "acme/claude-marketplace" is linked to "Acme"
- **When** Tom publishes "AWS management"
- **Then** a PR named "Packmind sync" is opened on "acme/claude-marketplace"

**Scenario: Published plugin shows as Packmind-managed**

- **Given** publish completes for "AWS management"
- **Then** marketplace screens show it as managed by Packmind, owner = space "Devops"

**Scenario: Successive publishes append to the same rolling PR**

- **Given** PR "Update plugins from Packmind" already has a "Security" commit
- **When** Tom publishes "AWS management"
- **Then** the same PR receives a new commit (no second PR is opened)

### Rule: Publishing is open to any org member _(new)_

**Scenario: Any member can publish a plugin**

- **Given** Alice is a member of the organization (not admin)
- **When** she publishes the plugin "Security"
- **Then** a PR is created on "acme/claude-marketplace"

### Rule: A publish without changes is a no-op _(new)_

**Scenario: Package with no content changes is not republished**

- **Given** package "Security" is already published
- **And** nothing has changed in it
- **When** Cédric publishes "Security"
- **Then** he is informed that no changes have been committed

**Scenario: Empty packages are still published (descriptor entry only)**

- **Given** package "Testing" containing nothing is added to "acme/marketplace"
- **When** it is published
- **Then** the package is added to the marketplace descriptor (no plugin content, descriptor entry only)

### Rule: Publish fails fast when contract or infra is broken _(new)_

**Scenario: Unmanaged plugin takes precedence over a managed one with the same name**

- **Given** a plugin "Security" exists in "acme/claude-marketplace" (unmanaged)
- **And** a package "Security" also exists in Packmind
- **When** Cédric adds "Security" and publishes it
- **Then** publishing is cancelled — the plugin already exists

**Scenario: Publish fails when the marketplace descriptor is missing**

- **Given** "acme/claude-marketplace" is linked but `marketplace.json` was removed from the repo
- **When** Cédric publishes
- **Then** publishing errors and the marketplace is marked `BAD_FORMAT`

**Scenario: Publish fails on expired Git token**

- **Given** the Git token has expired
- **When** Tom publishes "AWS management"
- **Then** publishing errors with message: "The package could not be published. Reason: Invalid or expired Git token"

## 7. Non-functional requirements

- **Concurrency**: two org members publishing simultaneously **must each get an independent PR**; PR opening runs as a background job to keep the UI responsive and to serialize Git operations on the marketplace repo. The marketplace descriptor carries a `packmind-lock` entry that records the canonical state Packmind expects on the next publish — used to detect drift and to make concurrent updates safe to merge in any order.
- **Performance**: a publish action returns control to the user within ~2s by enqueuing the background job; the resulting PR is visible on the marketplace repo within ~30s p95 under normal Git provider latency.
- **Security & privacy**: publishes use the marketplace-linked Git token; the token is never echoed back in errors or logs. The error surface explicitly enumerates the failure reason but never the secret.
- **Observability**: every publish emits a structured log line and an Amplitude event (see section 5). Failures are categorized so the success-rate dashboard can break them down by reason.
- **Accessibility**: any new UI surface (Publish button, status, error toast) meets WCAG 2.1 AA per PRODUCT.md — keyboard-reachable, visible focus, error programmatically associated with the field, color paired with text/icon.

## 8. Dependencies & risks

**Dependencies**

- Marketplace linking flow (admin-only) must already mark the marketplace as linked, with a valid Git token and a discovered `marketplace.json`.
- A background-job worker capable of running PR creation / amendment against the marketplace's Git provider (existing BullMQ infra per `CLAUDE.md`).
- The marketplace descriptor format must accommodate a `packmind-lock` field — descriptor schema change required. _(inferred — confirm)_

**Risks & mitigations**

- **Concurrent publishes diverge on the marketplace repo** → mitigated by background-job serialization and by `packmind-lock` carried in the descriptor (see section 7).
- **Token expiry surfaces deep in the job rather than at click-time** → fail-fast preflight check on token before enqueuing the job; preserve the explicit error message.
- **Descriptor schema migration breaks existing marketplaces** → ship the schema change with a forward-compatible default (`packmind-lock` absent = first publish), backfill on next publish.
- **Name collisions between Packmind packages and pre-existing unmanaged plugins** → cancel publish with a clear error (see Scenario in section 6); offer no automatic rename.

## 9. Rollout plan

- **Feature flag**: depends on marketplace feature flag
- **Phasing**:
  1. Descriptor schema change (`packmind-lock` field) + background-job scaffolding behind the flag, no UI exposure.
  2. Functional rule R2 (rolling PR + governance traces) + R-publish-access (any org member).
  3. R-publish-idempotency + R-publish-preconditions (error paths + no-op + empty packages).
  4. Telemetry wired (section 5 events), flag opened to beta orgs.
  5. GA after one full week without P0/P1 incidents on the publish path (after self using it internally)
- **Rollback**: flag flip disables Publish in the UI and stops the worker from picking up new jobs; in-flight jobs drain. No data migration to reverse since the descriptor change is additive.
- **Communication**: internal announce in the `#packmind` Slack channel; external release note when GA. _(inferred — confirm)_
