## User story

As an **org admin**,
I want **to grant Packmind access to specific Git repos under a clearly-named connection**,
so that **I can confidently tell which connection is used for what, and revoke or rotate the right one when needed**.

## Rule: A connection has a user-editable display name, unique within the organisation

Display name is the user-facing label that disambiguates connections of the same vendor. It is independent of the underlying identifier (org slug, instance URL, account login).

### Scenario: Creating a connection with an empty display name uses a placeholder in the list

- **Given** the org admin completes the connection install flow without entering a display name
- **When** the new connection appears in the list
- **Then** the display name column shows a placeholder (e.g. `Unnamed GitHub connection`) styled as muted/secondary text
- **And** the underlying stored value is the empty string, not the placeholder

### Scenario: Editing a connection's display name

- **Given** an existing connection with display name `Production marketplace`
- **When** the org admin opens its row actions and edits the display name to `Marketplace (prod)`
- **Then** the new value is persisted and the list reflects it immediately
- **And** the value is trimmed of leading/trailing whitespace before validation and persistence

### Scenario: Display name has a max length of 64 characters

- **Given** the display name edit dialog is open
- **When** the org admin types or pastes a string longer than 64 characters
- **Then** the input is capped at 64 characters with a visible counter
- **And** the save action explains the limit if a paste attempt exceeded it

### Scenario: Display name must be unique within the organisation (case-insensitive, trimmed)

- **Given** the org already has a connection named `Recipes` (vendor: GitHub)
- **When** the org admin tries to save a second connection (vendor: GitLab) with display name `Recipes` (or `recipes`)
- **Then** save is rejected with an inline error explaining the name is already used by another connection in this organisation
- **And** the existing connection is unaffected

### Scenario: Empty display name does not collide with itself

- **Given** two existing connections with an empty display name
- **When** the page renders
- **Then** both display the placeholder side by side without triggering the uniqueness error (uniqueness applies to non-empty values only)

## Non-functional notes

- Display-name edits emit an audit log entry (`actor_id`, `connection_id`, `old_value_present`, `new_value_present`, `at`).
- The edit dialog is keyboard-reachable, focus-trapped, and announces validation errors via `aria-live`.
- The rename is **UI only** — backend keeps the `provider` vocabulary.

## Out of scope

- Allowing rename on CLI-managed entries (see CLI-managed read-only tab issue).
