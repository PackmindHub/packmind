## User story

As an **org admin**,
I want **to see the health and last push date of each connection**,
so that **I can spot expired tokens or stale connections before they break a publish**.

## Rule: Each connection shows a health status and the date of the latest Packmind → repo push

Health and recency are surfaced directly on the connection row so users can audit the list without drilling into each connection.

### Scenario: A healthy connection is clearly marked as such

- **Given** a connection whose stored token is valid and whose last reachability check succeeded
- **When** the row renders
- **Then** a "Connected" status indicator is shown using both a label and an icon (no colour-only signal)
- **And** the latest push date column shows the timestamp of the most recent Packmind-initiated push (any artifact: standard, recipe, plugin, marketplace publish) made via this connection
- **And** if no push has ever happened via this connection, the column shows a `—` or `Never used` placeholder

### Scenario: A connection with an expired or invalid token surfaces the failure

- **Given** the stored credential for a connection has expired or been revoked vendor-side
- **When** the row renders
- **Then** the status indicator reads "Token expired" (or equivalent) with a warning icon
- **And** a row-level action lets the admin reconnect / rotate

### Scenario: A connection whose vendor API returned an error surfaces it as unreachable

- **Given** the last reachability check returned a non-auth error (network failure, rate limit, vendor 5xx)
- **When** the row renders
- **Then** the status indicator reads "Unreachable" with an error icon
- **And** a tooltip or details affordance exposes the underlying error class (without leaking secrets)

### Scenario: The status set is closed and explicit

- **Given** any user-configured connection
- **When** the status is computed
- **Then** it resolves to exactly one of: `Connected`, `Token expired`, `Unreachable`
- **And** statuses like "unknown" or "never checked" are not exposed in the UI (the backend resolves to one of the three before render)

### Scenario: On-demand refresh

- **Given** the org admin is viewing the connection list
- **When** they trigger the "Refresh" affordance on a row (and the list also pings on page load)
- **Then** the status is re-resolved and the row updates

## Non-functional notes

- Health and last-push timestamps are read from already-stored state — **no synchronous vendor API call on page load**. Use a background job + SSE emitter to surface information to the user.
- Last push date aggregates across artifact types (standards + recipes + marketplace publishes) into a single timestamp.
- Connection health resolution emits a structured log on each state transition with `org_id`, `connection_id`, `from_status`, `to_status`, and the sanitised reason.
- "Unreachable" tooltip must not leak token material or vendor response bodies; it surfaces a sanitised error class only.
- Every status/badge uses text + icon — never colour alone (WCAG 2.1 AA).
