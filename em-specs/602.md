## Context

The `settings/git` page should immediately communicate "this is where you give Packmind access to repos", not "here are some abstract providers". The "Connections" tab label has already been introduced — this issue completes the rebrand and the visual cues.

## Rule: Vocabulary and visual cues clarify the page's purpose

### Scenario: The page title and lead-in copy describe the user-facing purpose

- **Given** the org admin lands on `settings/git`
- **When** the page renders
- **Then** the page title reads in terms of _Git connections_ (not "Providers")
- **And** a short lead-in paragraph explains, in one sentence, that connections grant Packmind access to specific repos so it can publish standards, recipes, or marketplace plugins on the user's behalf

### Scenario: Vendor identity is reinforced visually on each row

- **Given** a list of connections mixing GitHub, GitLab (and any other supported vendor)
- **When** the list renders
- **Then** each row prefixes the connection with the vendor's official icon
- **And** vendor name is also rendered as text so the cue is not colour/icon-only (accessibility)

## Scope

- UI-only rename "providers" → "connections" across page title, breadcrumbs, headings, buttons, toasts, empty states, and any UI route segments (frontend routes only — backend stays as-is).
- Vendor icons (`SiGithub`, `SiGitlab`, …) prefixed on each row in `ConnectionsTable` and the CLI-managed table.
- Lead-in paragraph at the top of the page.

## Out of scope

- Backend rename `provider` → `connection` in code / DB / API contracts (tracked separately as a cleanup PRD).
