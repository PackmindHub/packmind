#### [T1] Linking transport / GitHub App with org-level OAuth scopes and webhook

**Status**: Not Covered
**What is missing**: The marketplace link/unlink implementation does not ride on a GitHub App installation flow. In the scoped implementation, admins select an existing connected Git provider and the backend requires `hasToken`; no marketplace-specific GitHub App installation, org-level OAuth scope carry-through, or webhook registration path was found.
**Where to look**: `apps/frontend/src/domain/marketplaces/components/PrivateLinkForm.tsx`, `apps/frontend/src/domain/marketplaces/components/LinkMarketplacePanel.tsx`, `packages/deployments/src/application/useCases/linkMarketplace/linkMarketplace.usecase.ts`, `apps/api/src/app/organizations/marketplaces/marketplaces.controller.ts`
**How to reproduce**:

1. Open the organization marketplaces page as an admin.
2. Start "Link a marketplace"; the UI asks for an existing Git provider plus owner/repo/branch, not a GitHub App installation or organization app authorization.
3. Submit using a tokenless provider; `LinkMarketplaceUseCase` rejects it via `GitProviderMissingTokenError` because `gitProvider.hasToken` must be true, whereas the spec expects the link/unlink flow to ride on GitHub App transport.

## Code Review

### Findings

#### [MEDIUM] Concurrent marketplace links can bypass duplicate rejection

- **Category**: Edge Case
- **File**: `packages/deployments/src/application/useCases/linkMarketplace/linkMarketplace.usecase.ts:136`
- **Description**: Duplicate detection is a pre-flight read before persistence. Two concurrent requests for the same organization/owner/repo can both observe no existing repo, then each creates a new marketplace-typed `GitRepo` at `packages/deployments/src/application/useCases/linkMarketplace/linkMarketplace.usecase.ts:199`. The marketplace table uniqueness only protects `(organization_id, git_repo_id)`, so distinct GitRepo ids for the same coordinates are not rejected. This can leave the same marketplace linked twice instead of returning the contract duplicate error.
- **Spec Reference**: Rule 1 / Example 4; Technical Rule: "Duplicate-link error message is part of the contract"
- **Suggested Fix**: Add a database-level uniqueness constraint or transactional upsert for active marketplace GitRepo coordinates within an organization, then map the resulting conflict to `MarketplaceAlreadyLinkedError`.

---

#### [MEDIUM] Unlink can leave marketplace cleanup half-applied

- **Category**: Edge Case
- **File**: `packages/deployments/src/application/useCases/unlinkMarketplace/unlinkMarketplace.usecase.ts:80`
- **Description**: The use case soft-deletes the marketplace row before deleting the underlying marketplace GitRepo. If `gitPort.deleteGitRepo` throws at `packages/deployments/src/application/useCases/unlinkMarketplace/unlinkMarketplace.usecase.ts:86`, the marketplace disappears from listings but the GitRepo row remains active. A later relink of the same owner/repo can then be rejected as already linked by the link pre-flight check, while the marketplace is no longer visible to admins.
- **Spec Reference**: Rule 3 / Example 1; Check Also: "Packmind-owned Git repo configuration is cleaned up"
- **Suggested Fix**: Perform marketplace and GitRepo soft-deletes in one transaction where possible, or delete/validate the GitRepo first and only remove the marketplace once cleanup succeeds. Add a failure-path test covering `deleteGitRepo` throwing.
